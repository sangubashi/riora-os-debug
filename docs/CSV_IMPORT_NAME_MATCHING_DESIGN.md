# CSV Import 顧客名寄せ設計（customer_hash_id 生成方式の見直し）

> **⚠️ SUPERSEDED (2026-06-20)**: `customers.customer_hash_id`前提の旧設計。`brain_*`方針確定により**廃止**（`brain_customers`の名寄せキーは`external_key_hash`等、別途`CSVImportSecurityArchitecture.md`で再設計）。本書は経緯記録としてのみ残置。

- 作成日: 2026-06-19
- 背景: 実CSVサンプル確認の結果、「お客様番号」列は存在するが実データが空欄であることが判明。従って`customer_hash_id`は番号ベースではなく**氏名ベースでの運用を継続**する。氏名ベース運用の精度を上げるため、表記ゆれ対策を設計する。
- 位置づけ: **設計のみ。実装は行わない。**

## 1. SalonBoard CSV 顧客名の表記ゆれ調査

リポジトリに実CSVサンプルファイルは存在しないため（`find . -iname "*.csv"`で確認済み、ヒット無し）、既存実装コード（`salonBoardParser.ts`/`normalizer.ts`）が対応済みのパターンと、**対応漏れしているパターン**を切り分けて調査した。

### 1.1 既に対応済みの表記ゆれ（`normalizer.ts`で処理済み）

| パターン | 例 | 対応関数 |
|---|---|---|
| 全角英数字 ↔ 半角英数字 | `Ｔａｎａｋａ` → `Tanaka` | `toHalfWidth()` |
| 全角スペース・タブ混在 | `田中　花子` / `田中\t花子` | `normalizeCustomerName()` |
| 連続スペース | `田中　　花子` | `normalizeCustomerName()`（`\s+`→単一スペース） |
| 前後の余分なスペース | ` 田中花子 ` | `normalizeCustomerName()`（`trim()`） |
| 敬称の有無 | `田中花子様` / `田中花子さん` / `田中花子` | `normalizeCustomerName()`（末尾の様/さま/さんを除去） |

### 1.2 未対応の表記ゆれ（要注意・現状リスクとして残る）

| パターン | 例 | 現状の扱い | リスク |
|---|---|---|---|
| ひらがな/カタカナの混在 | `たなか はなこ` vs `タナカ ハナコ` | `normalizer.ts`に`toNameKey()`という変換関数（ひらがな→カタカナ統一）は存在するが、**`SalonBoardImportEngine.ts`の`nameToKey()`はこれを使っていない**（後述2章で詳述） | 同一人物が別の来店時に表記が変わると別人として2件登録される |
| 半角カタカナ ↔ 全角カタカナ | `ﾀﾅｶ ﾊﾅｺ` vs `タナカ ハナコ` | `toHalfWidth()`は英数字・スペース・一部記号のみ対象で、**半角カタカナの正規化は行っていない** | POSレジ入力やCSV出力経路によって半角カタカナが混入するケースで名寄せ漏れ |
| 漢字の異字体（旧字体/新字体・人名漢字の表記差） | `渡辺` / `渡邊` / `渡邉`、`斉藤` / `斎藤` / `齋藤` | 異字体辞書・変換ロジックは**一切存在しない** | 同一人物でも別人として扱われる。フリガナを補助キーにすることで緩和可能（§3で設計） |
| 姓名の間のスペース有無・全角/半角スペースの違い | `田中花子` vs `田中 花子` vs `田中　花子` | スペース自体は§1.1で吸収されるが、`toNameKey()`は完全にスペース除去する一方`nameToKey()`は`\s+`削除のみ（結果は同じだが実装が分岐している点に注意） | 現状は実質問題なし（両関数とも最終的に空白を除去） |
| 改姓（結婚・離婚等によるフルネーム変更） | `佐藤花子` → `鈴木花子` | 名前ベースの正規化では原理的に解決不可能 | 同一人物が別顧客として新規登録される。電話番号等の不変識別子を保存しないPII方針上、現状この問題を完全に解決する手段はない。フリガナや来店パターン（担当者・施術内容の類似性）を使った人手レビュー支援が現実的な対応 |
| 同姓同名の別人 | 本当に別人の「田中花子」が2人いる場合 | 現状の氏名ベースキーは**同一人物として誤統合するリスク**を本質的に抱えている | 番号データが空欄である限り構造的に避けられない。重大な誤統合を避けるため、自動マージは行わず候補検出＋人手確認に留めるべき（既存`detectDuplicateCustomers()`の方向性） |

## 2. customer_hash_id 生成前の正規化設計

### 2.1 現状の実装フロー（再確認）

```
CSV生データ "田中　花子様"
   │
   ▼ salonBoardParser.ts 内で呼び出し
normalizeCustomerName()
   │  toHalfWidth() → trim() → 全角スペース/タブ統一 → 連続スペース圧縮 → 敬称除去
   ▼
"田中 花子"  ← SalonBoardRawRow.customerName に格納される値
   │
   ▼ SalonBoardImportEngine.ts 内で呼び出し
nameToKey()
   │  trim() → \s+ 除去 → 全角スペース除去 → djb2ハッシュ
   ▼
"cust_xxxxx"  ← customers.customer_hash_id に保存される値
```

**問題点**: `nameToKey()`は`normalizer.ts`の`toNameKey()`（ひらがな→カタカナ統一を含む、最も厳密な照合キー生成関数）を使わず、自前で簡易な正規化（空白除去のみ）を再実装している。結果として、すでに用意されている「ひらがな/カタカナ統一」ロジックがハッシュキー生成の段階で活用されていない。

### 2.2 設計案（実装はしない・方針のみ）

`nameToKey()`の入力を、独自の簡易正規化ではなく`toNameKey()`の出力に統一する。

```
変更前: nameToKey(name) = djb2( name.trim().replace(/\s+/g,'').replace(/　/g,'') )
変更後: nameToKey(name) = djb2( toNameKey(name) )
                          ※ toNameKey() = normalizeCustomerName() + ひらがな→カタカナ + 空白完全除去
```

この変更により、ハッシュ生成前に以下が確実に統一された状態になる：

1. 全角英数字 → 半角（`toHalfWidth()`）
2. 前後スペース削除（`trim()`）
3. 連続スペース除去（`normalizeCustomerName()`内の`\s+`圧縮、`toNameKey()`では最終的に空白を完全除去）
4. ひらがな → カタカナ統一（`toNameKey()`独自処理）
5. 敬称除去（`normalizeCustomerName()`）

**半角カタカナの統一は現状`toNameKey()`にも実装されていないため、別途追加が必要**（§1.2の指摘事項）。設計としては`toHalfWidth()`相当の処理に半角カタカナ→全角カタカナ変換テーブルを追加するか、`toNameKey()`内で専用の変換ステップを挟む形を想定する。実装時の詳細（変換テーブル方式 or 既存ライブラリ利用）は次フェーズで検討。

### 2.3 影響範囲

- `nameToKey()`の正規化ロジックを変更すると、**既存にすでに保存済みの`customer_hash_id`値と不一致が生じる**（ハッシュアルゴリズム自体は変えないが、入力文字列の正規化方法を変えるため、同じ氏名でも生成されるハッシュ値が変わる可能性がある）。これは実装時に「既存`customers`データの`customer_hash_id`再計算・一括更新」が必要になることを意味する。マイグレーション要否を含めて実装フェーズで別途設計する。

## 3. フリガナを名寄せ補助キーとして利用する設計案

### 3.1 前提確認

- `customers.name_kana`列は**実DBに既に存在する**（`DB_AUDIT_REPORT.md`確認済み、text、NULL許可）。フリガナ保存先として新規カラム追加は不要。
- 現状の`SalonBoardColumnMap`/`COLUMN_ALIASES`には**フリガナ列のエイリアスが定義されていない**。CSVにフリガナ列が存在しても現状は読み取られない。

### 3.2 設計方針

#### (a) フリガナ列の検出を追加（任意項目として）

`COLUMN_ALIASES`に`customerKana: ['フリガナ', 'ふりがな', 'カナ', 'kana']`を追加する想定。フリガナ列がCSVに存在しない場合も既存の必須カラムチェック（顧客名・来店日）には影響しないよう、**任意項目**として扱う（無くてもエラーにしない）。

#### (b) フリガナの正規化

取得したフリガナ値に対し、`toHalfWidth()` + ひらがな→カタカナ統一を適用し、`customers.name_kana`へ保存する正規化済み値を作る。

#### (c) フリガナの使い道を2段階に分離する

| 用途 | 採用するか | 理由 |
|---|---|---|
| ① `customers.name_kana`への保存（表示・検索用） | **採用** | 既存カラムを活用するだけで、リスクのない単純な保存 |
| ② `customer_hash_id`生成キーへフリガナを直接組み込む（氏名+フリガナで1つのハッシュを作る） | **不採用** | フリガナが無い来店データ（フリガナ列が空の行）が混在すると、同一人物でもハッシュが変わってしまい、却って名寄せ精度が落ちる |
| ③ 氏名ベースの`customer_hash_id`が一致しない場合の**候補検出**（自動マージはしない） | **採用（人手確認前提）** | 異字体の同一人物候補をフリガナの一致で見つけられる（例: 渡辺花子 / 渡邊花子 → ともにフリガナ「ワタナベハナコ」なら候補として提示）。ただし同姓同名・同フリガナの別人を誤統合するリスクがあるため、**自動マージは行わず、既存`detectDuplicateCustomers()`の出力形式を拡張し、運用者が確認・手動マージ判断する仕組みに留める** |

#### (d) 既存`detectDuplicateCustomers()`との関係

現行の`detectDuplicateCustomers()`は氏名の`toNameKey()`一致のみで重複候補を検出している。フリガナ情報が使えるようになった場合、検出条件を「氏名キー一致 **または** フリガナキー一致（かつ氏名キー不一致）」に拡張することで、異字体パターンを候補として拾えるようになる。この拡張も実装フェーズで設計する（本書では方針のみ）。

### 3.3 結論

フリガナは「保存先カラムが既にある」「異字体問題の緩和に使える」という点で有用だが、**ハッシュキーそのものに混ぜるのは精度を下げるリスクがあるため不採用**。「自動マージ」ではなく「候補提示→人手確認」の補助情報として位置づけるのが安全な設計と判断する。

## 4. まとめ・次フェーズへの持ち越し事項

1. `nameToKey()`を`toNameKey()`ベースに統一する変更は、既存データのハッシュ再計算を伴うため、実装時に移行手順（既存`customers.customer_hash_id`の再計算バッチ）を別途設計する必要がある
2. 半角カタカナ → 全角カタカナの統一処理は現状どこにも実装されていない。追加するかどうか、実際のCSVサンプルでの出現有無を確認した上で判断するのが望ましい（本書はコードを変更しないため確認は次フェーズ）
3. フリガナ列のCSV取込対応（`COLUMN_ALIASES`拡張・`customers.name_kana`への保存）と、`detectDuplicateCustomers()`の検出条件拡張は、いずれも設計のみ完了。実装は別途承認後に着手する
4. 同姓同名の別人を誤統合するリスクは氏名ベース運用の構造的な限界であり、「お客様番号」列に実データが入るようになれば本質的に解消される。CSV出力設定側でお客様番号を有効化できないか、SalonBoard側の運用確認も並行して検討する価値がある（本書の調査範囲外）
