## バグA: LINE反応率が常に0%固定

- **深刻度**: 中
- **場所**: `src/components/phase1/Phase1Screen.tsx` の `toCustomer()`（`line_response_rate: 0`
  がハードコード）
- **影響範囲**:
  - `CustomerBottomSheet.tsx` のKPI横3列「LINE反応率」タイルが常に `0%` と表示される
    （事実情報のはずが実データではなく固定値）
  - `NextActionPanel.tsx` の `line_low_response` ルール（`lineResponseRate < 60` で発火）が
    常に条件を満たしやすくなり、AI提案の精度に副作用がある
  - `CustomerRiskCard.tsx` の内部リスク評価にも同じ固定値が渡る
- **発生時期**: v1.0リファクタ以前から存在（今回新規に作り込んだものではない）
- **想定される修正方針**（v1.0.1で検討）: `line_send_logs` 等の実データから応答率を算出する
  クエリを `useHomeStore` / `brain-stats` API 経路に追加する

## バグB: 音声メモのマイク許可タイムアウトタイマーが解放されない

- **深刻度**: 中
- **場所**: `src/hooks/useVoiceRecorder.ts` の `start()`（15秒タイムアウト用 `setTimeout`）
- **内容**: `getUserMedia` が15秒以内に成功した通常ケースでは、タイムアウト用の `setTimeout` が
  クリアされないまま残り続ける。録音セッションが15秒を超えて継続した後、同一コンポーネントで
  再度 `start()` が呼ばれるタイミングが重なると、孤立したタイマーのコールバックが
  `timedOutRef.current` を書き換え、新しい録音開始が無言で失敗する可能性がある
- **影響範囲**: `VoiceMemoSection.tsx` からの録音操作（発生頻度は低い・特定のタイミングでのみ
  再現するレースコンディション）
- **発生時期**: v1.0リファクタで追加したタイムアウトフォールバック機能自体に内在する新規バグ
- **想定される修正方針**（v1.0.1で検討）: `setTimeout` のIDを保持し、`Promise.race` 確定後
  （成功・失敗いずれの場合も）に `clearTimeout` する

## 既知課題C: デモ環境で「本日の予約0件」が発生する（バグではなく運用課題）

- **深刻度**: 低（本番では発生しない想定のため優先度を下げて記録）
- **性質**: アプリ側のバグではなく、デモ環境の予約データ運用に起因する既知課題
- **内容**: `reservations.scheduled_at` の「今日データ」が、固定日付の手動SQL投入でしか
  作られておらず、自動的なロールフォワードの仕組みがない。投入した当日はホーム画面に
  予約が表示されるが、翌日以降は`/api/home/reservations`のtoday判定に該当する行が
  1件も無くなり「本日のお客様 0件」と表示される。API・today判定・JST変換・staff_id認証
  フィルタのロジック自体は正しく動作していることを確認済み（詳細は2026-07-03付
  「本日のお客様 0件」原因調査レポート・「デモ環境運用ルール」調査レポート、会話履歴参照）
- **対応方針**: **v1.0では対応しない。** 本番運用では実際の予約が継続的に投入されるため
  発生しない問題であり、優先度が低いと判断。デモ・検証時に必要であれば、手動での
  データ再投入（当日日付への差し替えSQL実行）で都度対応する
- **参考として検討した仮対応案**（いずれも未実装）: (A) 手動ロールフォワードSQLの整備、
  (B) reservationsシードを`CURRENT_DATE`基準の相対日付に書き換え、(C) 検証用cronでの
  自動複製、(D) 0件時に「最終更新日」を明示するフォールバックUI表示
- **本番運用時**: この課題自体が本質的に発生しなくなるため、対応不要（`docs/V2_HANDOFF_NOTES.md`
  参照）
