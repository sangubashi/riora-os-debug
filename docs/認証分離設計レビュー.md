## 1. DEMO_MODEを完全撤去するために必要な変更

`DEMO_MODE`単体を`false`にするだけでは§0の通り認証が壊れる(前回調査で確認済み)。完全撤去には以下が必要。

### 1-1. `app/ClientShell.tsx`本体の再設計
- 自動サインインEffect(`:78-91`)の`if (!DEMO_MODE) return`を撤去し、**自動サインイン自体を「デモ環境専用の機能」として環境変数に置き換える**(本番では存在しない経路にする。詳細は§2)
- `PUBLIC_PATHS`(`:17-28`)から`/customers`/`/kpi`/`/line`/`/menu`/`/ai-suggestions`/`/phase1`を削除し、認証必須パスに戻す。`/phase1-debug`/`/test`は本番ビルドそのものから除外する判断が必要(後述)
- ログインリダイレクトEffect(`:94-105`)の`if (DEMO_MODE) return`を撤去し、常時有効化する

### 1-2. lib層・store層の個別フラグ分岐(49ファイル)
前回調査で列挙した全箇所(`if (DEMO_MODE)`/`if (DEMO_MODE && !VOICE_NOTES_LIVE)`)について、各ファイル単位で以下のいずれかを選択する必要がある:
- **(a) モック分岐を削除し、実Supabase経路のみ残す**(`voiceNote.ts`/`contraindication.ts`/`handover.ts`/`bookingPrompt.ts`/`customerNotes.ts`等、false分岐が完成しているファイル)
- **(b) テーブル名の整合性確認後に(a)を適用**(`useHomeStore.ts`の`kpi_today`、`useKpiSqlStore.ts`の`line_logs`など、実在未確認のテーブルを参照する箇所)
- **(c) 新規実装してから(a)を適用**(`OccupancyHeatmap`/`RepeatAnalytics`/`KpiDashboard`の`AI_STRATEGY`、AI提案画面全体など、Supabase接続フック自体が無い箇所)
- **(d) 配線(呼び出し元のuseEffect等)を追加してから(a)を適用**(`useMenuStore`の`fetchMenus`等)

### 1-3. 矛盾・非対称の解消(撤去前に必ず修正すべき項目)
- `useCustomerStore.ts`/`useHomeStore.ts`の`isMockMode()`(`DEMO_MODE && !VOICE_NOTES_LIVE`)と、同ファイル内の他の`if (DEMO_MODE)`単体チェックが異なる基準で混在している。撤去時にどちらの基準で統一するか決める必要がある
- `aiMemory.ts`が`VOICE_NOTES_LIVE`を見ていない(他のVoice系ファイルと非対称) → 撤去前に意図を確認(本当にVoiceNotes Liveと連動すべきか)
- `actionLog.ts`の書込(`logAction`)と読込(`fetchRecentActions`)でフラグ判定基準が異なる(保存したのに一覧に出ない非対称)

### 1-4. 死んだコード・孤立コンポーネントの扱い決定
DEMO_MODE撤去とは直接関係ないが、撤去作業中に必然的に触れることになるため、このタイミングで方針を決めるべき:
- `src/components/LoginScreen/index.tsx` + `src/lib/auth.ts`(誰からもimportされない競合ログインUI) → 削除推奨
- `src/components/StaffTopScreen/`(誰からもimportされない旧ホーム画面) → 削除推奨
- `StaffKpiCard.tsx`/`StaffRanking.tsx`/`AIInsightBox.tsx`/`AISuggestionCard.tsx`(2種) → 配線して活用するか削除するかの判断
- `MenuDashboard.tsx`の死んだ「設定」タブ → 削除または実装

## 2. DEMO環境と本番環境の分離設計

### 現状の問題
`DEMO_MODE`は`src/lib/supabase.ts`にハードコードされた`true`であり、**環境変数ではない**。デモ用と本番用でビルドを分ける仕組みが存在せず、コードを書き換えてデプロイし直す以外に切り替え手段が無い。さらに、デモ環境と本番環境は**同一のSupabaseプロジェクト**(`ohszxgajckzphhfhdrsv.supabase.co`)を共有している(本セッションの全タスクで確認済み)。

### 推奨する分離設計

| 項目 | 現状 | 推奨 |
|---|---|---|
| フラグの保持方法 | `src/lib/supabase.ts`の定数 | `process.env.NEXT_PUBLIC_DEMO_MODE`(Vercel環境変数。Production未設定=false、Demo用Preview/別プロジェクトのみ`true`) |
| デプロイ先 | 単一のVercelプロジェクト | 本番用プロジェクトと、デモ用Vercelプロジェクト(またはPreview環境)を分離。デモ用のみ`NEXT_PUBLIC_DEMO_MODE=true`を設定 |
| データベース | 単一のSupabaseプロジェクトを共有 | デモ用は別のSupabaseプロジェクト(または最低限、本番と物理的に分離されたschema/store)を用意。理由: モック分岐が削除された後、デモ環境での自動サインイン(`admin@salon-riora.jp`)が誤って本番データに書き込むリスクを完全排除するため |
| デモの中身 | コード内に散らばった`MOCK_*`配列 | (任意・将来検討)ハードコード配列ではなく、デモ専用Supabaseプロジェクトに投入したシード データを使う設計にすれば、デモ体験の質も上がり保守コストも下がる。ただし本タスクの必須要件ではない |

## 3. スタッフ側ログインフロー

### 現状(実装済みで動作確認済みの経路)
`app/login/page.tsx` → `useAuthStore.signIn({email, password})` → `supabase.auth.signInWithPassword()` → セッション確立 → `/phase1`へ`router.push`。

このフローは本セッション内のPlaywright実機テストで`admin@salon-riora.jp`/`riora2026`によるログイン成功を**既に確認済み**(MD-6完成レポート作成時の実機検証時)。

### 設計上の課題
1. **ロールがstaff側のグローバル状態に載っていない**: `useAuthStore`は`user`/`session`は保持するが`role`を保持していない。`useCustomerStore.fetchCustomers()`内で都度`profiles`から`role`を取得しているなど、ロール取得が画面ごとに個別実装されており一元化されていない。DEMO_MODE撤去後、AppBottomNav等でロールに応じた表示制御をしたい場合、ログイン直後に`profiles.role`を取得し`useAuthStore`(または新設の`useRoleStore`)に保持する設計が必要
2. **招待ベースのオンボーディング(`useAuthStore.signUp`/`inviteStaff`/`fetchInvitations`)が実際に動くか未検証**: `check_signup_allowed()`RPCに依存する設計だが、本調査ではこのRPCの存在・動作を確認していない。新規スタッフのオンボーディング手順として採用する前に別途検証が必要
3. **死んだ競合UI(`LoginScreen/index.tsx`)の削除**(§1-4で前述)

### 推奨フロー(変更なしでよい部分)
ログインのコア機構(`app/login/page.tsx`)自体は完成しており、DEMO_MODE撤去後もそのまま利用できる。変更が必要なのは「ログインを強制する仕組み」(ClientShellのガード)の側であり、ログイン画面自体の再実装は不要。

## 4. 管理者側ログインフロー

### 現状: 認証・認可が一切存在しない(最重要課題)

`app/admin/*`(MD-1経営TOP〜MD-6 CSV Import、および統合レイアウト)には:
- `app/admin/layout.tsx`にセッションチェックが無い
- 各`app/api/admin/*`ルートが`getRepos()`経由で**SERVICE_ROLE_KEY**を使用し、RLSを完全にバイパスする
- ルート内でのsession/role検証コードが一切無い(CSV Import完成レポート以降、毎回「既知のギャップ」として記録されているが未着手)

これは**DEMO_MODEとは無関係の、独立した未解決のセキュリティギャップ**である。DEMO_MODEを撤去しても、この問題は自動的には解決しない(admin配下のコードはDEMO_MODEを一度もimportしていない)。

### 設計上の要点
1. **RLSでは対応できない**: SERVICE_ROLE_KEYはRLSを意図的にバイパスする設計のため、認可チェックは**アプリケーションコード側**(各Route Handler、またはNext.js Middleware)で行うしかない
2. **チェックすべき内容**: (a)有効なSupabaseセッションが存在するか、(b)`profiles.role`が許可されたロール(owner、場合によりadmin/manager)であるか。この2点を`getRepos()`呼び出し前に必ず検証する
3. **実装方式の選択肢**:
   - Next.js Middleware(`middleware.ts`)で`/admin/*`への全リクエストを横断的にガードする(実装箇所が1か所に集約できる利点)
   - 各Route Handlerの先頭で個別にチェックする関数を呼ぶ(柔軟だが各ファイルへの追記が必要、漏れのリスク)
   - `app/admin/layout.tsx`(クライアント側)でのリダイレクトガードのみでは**不十分**(APIルート自体は直接URLを叩けば素通りするため、UIだけのガードはセキュリティ対策にならない)
4. **`app/admin/layout.tsx`のUI側ガードも別途必要**: API側を保護した上で、UI側(`layout.tsx`)にも未ログイン/権限不足時のリダイレクトを追加し、UXとして自然に弾く

## 5. ロール(owner/admin/staff)設計

### 発見した不整合: ロール名が2系統で矛盾している

| 定義箇所 | ロール値 |
|---|---|
| DB: `profiles.role` CHECK制約(`20250515000008_strict_rls.sql`等) | `'owner' \| 'admin' \| 'staff'` |
| TS型: `src/store/useDashboardStore.ts:37` `UserRole` | `'owner' \| 'admin' \| 'staff' \| null` |
| TS型: `app/lib/roles.ts` `INTERNAL_ADMIN_ROLES`/`InternalAuthRole` | `'owner' \| 'manager' \| 'staff'`(**`admin`ではなく`manager`**) |

`app/lib/roles.ts`は`'admin'`という値を一切認識せず、`getRoleDisplayName()`では`case 'admin': return '管理者'`という分岐もあるため部分的に認識しているが、`INTERNAL_ADMIN_ROLES`配列(権限判定の本体)には`'admin'`が含まれていない。**DBが`'admin'`を許容していても、`isAdminRole('admin')`は`false`を返す**(`INTERNAL_ADMIN_ROLES.includes('admin')`が偽になるため)。これは実害のあるロジックバグであり、DEMO_MODE撤去後にロールベースの認可を実装する前に**必ず解消すべき**。

### 設計判断が必要な点
1. 正式なロール名を`'admin'`と`'manager'`のどちらに統一するか(DB制約・`useDashboardStore`は`admin`、`app/lib/roles.ts`は`manager`。どちらかをリネームする必要がある)
2. 管理者ダッシュボード(`/admin/*`)へのアクセスを「owner専用」にするか「owner+admin(or manager)」にするか。各完成レポートの記述は概ね「owner専用」だが、v2.0アーキ文書では画面によって「管理者専用」という別の表現も使われており、用語の対応関係を明文化する必要がある
3. Brain側の`is_owner()`/`is_staff_or_owner()`(`20260618_staff_roles_setup.sql`)は`profiles.role`を参照する設計のため、上記1の決定はBrain側RLSにも影響する

### 既に統一されている良い点
`profiles.id` = `auth.users.id` = `brain_staff.auth_uid`という単一アイデンティティ設計は既に貫かれている(`20260618_staff_roles_setup.sql`のINSERT文で確認済み)。ロール名の不整合を解消すれば、ID統合自体は崩さずに済む。

## 6. RLSへの影響

### 6-1. 旧スキーマ(`customers`/`reservations`/`staff_logs`等)
`get_my_staff_id()`/`is_owner()`に基づくRLSは設計上は完成しているが、**これまで実運用での検証機会が乏しい**(DEMO_MODE分岐により多くの画面が実クエリに到達していなかったため)。DEMO_MODE撤去後、これらのポリシーが実際のログインセッションに対して正しく機能するかは個別に再検証が必要。

### 6-2. Brainスキーマ(`brain_customers`/`brain_visits`等)
**`brain_customers.assigned_staff_id`がCSV Import経由のどの顧客にも設定されていない**(本セッションのMD-3/MD-4タスクで確認済み・常にnull)。Brain側のstaff向けRLS(staffは`assigned_staff_id = auth.uid()`の行のみ参照可能、という設計が存在する場合)は、この列が空である限り**staffロールでログインしても0件**になる。DEMO_MODE撤去とは別に、(a)CSV Import側でassigned_staff_idを設定する仕組みを作る、または(b)staff向けRLSの条件を見直す、のいずれかが必要。

### 6-3. 管理者ダッシュボード
§4の通りSERVICE_ROLE_KEYがRLSを完全にバイパスしているため、**RLSは管理者ダッシュボードに対して無力**。認可はアプリケーションコードで担保するしかない。

### 6-4. 撤去作業全体への影響
モック分岐を外すたびに、その画面が初めて「本物のRLS」にさらされることになる。これまでDEMO_MODEによって隠れていたRLSポリシー漏れ・GRANT文不足・auth.uid()不一致が、画面ごとに順次表面化する可能性がある。一括でフラグを倒すのではなく、画面単位で1つずつ外して都度検証する方式(§7)が安全。

## 7. 段階的移行手順

```
Phase 1: 認証基盤の分離(最優先・他の全フェーズの前提)
  1-1. NEXT_PUBLIC_DEMO_MODE 環境変数を導入(Vercel側で本番=未設定/false、デモ環境=true)
  1-2. ClientShell.tsxの自動サインインを環境変数ベースに置き換え
  1-3. PUBLIC_PATHSを /login, /splash のみに縮小
  1-4. ロール名の不整合(admin/manager)を解消(§5)
  → この時点でステージング環境で「ログインしないと何も見えない」状態を作り、
    既存の主要フローが手動ログイン後に問題なく動くことを確認する

Phase 2: ロール読み込みの一元化
  2-1. useAuthStore(または新設ストア)にログイン後のrole取得を統合
  2-2. AppBottomNav等、ロール分岐が必要なUIを特定し対応方針を決定

Phase 3: 管理者ダッシュボードの認可追加
  3-1. /admin/* へのMiddleware(またはlayout.tsx+各Route Handler)で
       session+role検証を追加(owner専用 or owner+admin、Phase1-4の決定に従う)
  3-2. UI側(admin/layout.tsx)にも未認可時のリダイレクトを追加

Phase 4: 個別画面のDEMO_MODE分岐解除(完成度が高い画面から優先)
  4-1. 顧客詳細画面(CustomerBottomSheet等) → 解除→実機検証
  4-2. LINE画面(チャット系) → テーブル実在性確認→解除→実機検証
  4-3. KPI画面 → テーブル名不一致(kpi_today/line_logs)解消→解除→実機検証
  4-4. メニュー画面 → 呼び出し配線追加→解除→実機検証
  (AI提案画面・設定画面は新規実装が前提のため本フェーズの対象外)

Phase 5: RLS実地検証
  5-1. owner/admin/staff それぞれの実アカウントでログインし、各画面のデータが
       想定通り表示・制限されることを確認
  5-2. assigned_staff_id未設定問題への対応(§6-2)

Phase 6: 死んだコード・矛盾の削除
  6-1. LoginScreen/index.tsx, src/lib/auth.ts, StaffTopScreen等の削除
  6-2. isMockMode()等の重複フラグ判定ロジックの整理
  6-3. DEMO_MODE / VOICE_NOTES_LIVE 変数自体の削除(最後)

Phase 7: デモ環境の再構築
  7-1. 本番から独立したデモ用Vercelプロジェクト+(可能であれば)別Supabaseプロジェクトを用意
  7-2. NEXT_PUBLIC_DEMO_MODE=true をデモ環境のみに設定し、Phase1で整理した
       自動サインイン経路をそこでのみ有効化
```

**Phase 1が完了するまで、Phase 4以降のいずれの画面のフラグも個別に外してはならない**(§0の通り、認証基盤が分離されていない状態で個別フラグだけ外すと、未認証状態でのSupabase呼び出しが発生し、RLSエラー/空表示という新たな不具合を生むため)。
