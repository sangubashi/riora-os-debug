## 1. Androidでのインストール導線確認

本番相当ビルド(`npm run build` + `npm run start`)に対し、Android Chrome の
UserAgentを設定したPlaywrightで、実際にログインした状態で確認した。

### 1-1. 静的なインストール成立要件（自動確認済み）

```
manifest.json:
  name / short_name           あり
  start_url                    あり("/")
  display                      "standalone"
  icons 192×192(purpose:any)   あり
  icons 512×512(purpose:any)   あり
  icons 192×192(purpose:maskable) あり
  icons 512×512(purpose:maskable) あり

Service Worker:
  登録数                        1件
  現在のページを制御しているか   はい(controller有効)
```

Android/Chromeが「インストール可能」と判定するために必要な静的条件
(manifestの必須フィールド・アイコンの必須サイズ・Service Workerの登録)は
**すべて満たしていることを確認した。**

### 1-2. `beforeinstallprompt`自体の発火確認（限界あり）

前フェーズ(PWA-5)では`beforeinstallprompt`イベントを人為的に発生させて
UIの表示ロジックを検証したが、今回は**人為的な発火を行わず**、実際に
Chromeエンジンが自然にこのイベントを発火するかを確認した。

```
結果: 3秒待機しても自然発火しなかった
```

これは**バグではなく、テスト環境の既知の限界**と判断する。Chromeの
`beforeinstallprompt`は、manifest/Service Workerの条件を満たすだけでなく、
**「ユーザーがある程度の期間・頻度でサイトを訪れている」という
エンゲージメント判定**を内部的に行っており、これは自動化された
1回きりのヘッドレスアクセスでは再現できない(Google自身も
Lighthouse等のツールでこの判定を完全にシミュレートすることはできないと
明記している類のロジック)。

**そのため、UIの表示ロジック自体(PWA-5で検証済み)は正しく実装されているが、
実際にAndroid実機・実際のChromeで`beforeinstallprompt`が発火し、
「追加する」ボタンから本当にインストールが完了するかは、実機での
最終確認が必要。**

### 1-3. Android実機での確認手順（推奨）

```
1. Android端末のChromeで本番URLに複数回(2〜3回程度)アクセスし、
   多少の間隔を空けて再訪問する(エンゲージメント条件を満たすため)
2. ログイン後、Today画面(2回目以降の訪問)で「ホーム画面に追加しませんか？」
   という案内バーが表示されるか確認
3. 「追加する」をタップ → OS標準のインストールダイアログが出るか確認
4. インストール後、ホーム画面のアイコンから起動し、standalone表示に
   なっているか確認
5. 「あとで」を押した場合、7日間再表示されないことを確認(即時確認は
   困難なため、localStorageの riora_pwa_install_dismiss_until が
   未来の日時になっていることをdevtoolsで確認する形でも可)
```

## 2. iPhone実機想定の最終確認手順（これまでの内容を統合）

これまでの各フェーズ(PWA-2 アイコン・PWA-2c スプラッシュ・PWA-3 metaタグ・
PWA-5 インストール導線)の確認手順を1つに統合した、実機での通し確認手順。

```
□ ステップ1: 初回アクセス
  ・SafariでURLを開く。この時点ではインストール案内は出ない(初回非表示)
  ・アドレスバーの色がピンク系(#F56E8B)になっているか

□ ステップ2: ホーム画面に追加
  ・共有ボタン→「ホーム画面に追加」
  ・追加時のプレビューでアイコンがクリーム背景+Salon Rioraロゴになっているか
    (icon-192.png / apple-touch-icon.png)
  ・アイコン名が「Riora」になっているか

□ ステップ3: ホーム画面からの起動(最重要)
  ・追加したアイコンをタップして起動
  ・Safariのアドレスバー等が表示されず、フルスクリーンになっているか
    (apple-mobile-web-app-capable)
  ・起動直後、白画面のごく短い一瞬の後、リオくまのスプラッシュ画像が
    表示されるか(apple-touch-startup-image、対応機種の場合)
  ・スプラッシュが約0.8秒表示されてから、ログイン画面(未ログイン時)
    またはToday画面(ログイン済み時)へ切り替わるか
  ・ステータスバー(時刻等)がコンテンツに重なって見えるが、
    コンテンツの上端が隠れて読めなくなっていないか

□ ステップ4: 2回目以降の訪問でのインストール案内
  ・一度アプリを離れ、再度Today画面を開く(通常のブラウザ、またはPWA自体でも可)
  ・「ホーム画面に追加できます」という案内カードが表示されるか
  ・「あとで」を押すと消えて、その後7日間出ないか

□ ステップ5: LINE等アプリ内ブラウザからのアクセス(注意喚起の確認)
  ・LINEのトーク内リンク等からアプリを開く
  ・「この画面ではホーム画面に追加できません。Safariで開いてください」
    という案内が出るか(共有メニュー自体に「ホーム画面に追加」が
    出ない、iOS特有の制約に対する案内)

□ ステップ6: 設定タブでの常設ガイド確認
  ・設定タブ→使い方ガイド→「ホーム画面に追加する方法」を開き、
    iPhone/Android両方の手順が表示されるか
```

## 4. 残課題の一覧

```
1. 【機能未実装】PWA-4 オフライン専用画面
   現状、public/sw.jsはオフライン時に "/phase1" のキャッシュへ
   フォールバックするのみで、正本が想定する「オフラインです。
   電波の良い場所で〜」という専用の優しい案内画面は無い。

2. 【キャッシュ範囲の不足】sw.jsのSHELLキャッシュリストに、
   ルート"/"・"/login"・"/splash/splash-bg.png"・maskableアイコン・
   apple-startup画像が含まれていない。完全オフライン状態で
   PWAを起動した場合、スプラッシュ画像やログイン画面が
   表示されない可能性がある(APIはNetwork First方針で意図的に
   キャッシュしていないが、静的なシェル画像は含めた方が望ましい)。

3. 【実機未確認】Android実機での`beforeinstallprompt`実発火から
   インストール完了までの一連の流れ(本書§1-3の手順)。

4. 【実機未確認】iPhone実機でのスプラッシュ表示・standalone起動・
   インストール導線の一連の流れ(本書§2の手順)。

5. 【ツール未導入】Lighthouse CLI(npx lighthouse)によるPWA監査の
   形式的なpass確認。本フェーズではネットワーク経由の新規インストールを
   伴うため実施しなかった。

6. 【手動確認のみ】「3回あとでを押すと恒久非表示になる」ロジックは
   コードレビュー上正しいが、実際に3回連続発生させる長時間シナリオでの
   実機確認は未実施(PWA-5から継続する残課題)。
```
