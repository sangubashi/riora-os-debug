# Management Dashboard Architecture v2.0

**株式会社martylabo / Salon Riora — 管理者ダッシュボード 再構成版(確定)**
作成日: 2026-06-12
正典関係: v1.0(7画面)/v1.1(スタッフ表示制限)を**本書が統合・置換**する。経営レビュー(機能取捨選択)を反映。
編集方針: **「残す機能を優先順に厚く、削る機能はきっぱり外す」**。毎日見る画面を減らし、1画面の密度を上げる。

## 0. 取捨選択の結論(レビュー反映)

| 判定 | 機能 | v2.0での扱い |
|---|---|---|
| ★残す#1 | AIインサイト | **一行の行動指示**に限定(「◯◯様に炭酸パックを提案」)。気づき列挙は禁止。経営TOPのWarning+専用Actionリストの2層 |
| ★残す#2 | KPI集計 | **3〜5指標に厳選**。本日売上/今月目標進捗/次回予約率/DM→予約転換率。異常値だけ拾える |
| ★残す#3 | 失客 | **予兆アラート**に作り替え(事後分析でなく間に合う通知)=離脱予兆センター |
| ★残す#4 | スタッフ別売上 | 管理者専用。**売上単体禁止・指名率/リピート率と必ずセット**。スタッフ本人にランキング非表示(v1.1継承) |
| 削除 | カレンダー | SalonBoardが唯一の正。二重管理を作らない(元々非搭載) |
| 削除 | LINE分析タブ | **「DM→予約転換率」1指標だけ経営TOPへ昇格**。独立タブは消す |
| 格下げ | レポート | 常設タブから外し**Audit Report方式**(ボタン起動・ON/OFF・7日キャッシュ)へ委譲(別正典で実装済) |
| 非表示 | 店舗別比較/多店舗UI | store_idはスキーマ済。**UIは2店舗目が決まった月に初出**。今は出さない |
| ＋追加#1 | 次回予約率 | チェックアウト時リピート予約=retentionの先行指標。経営TOP KPI+スタッフ分析 |
| ＋追加#2 | 席稼働率 | 曜日・時間帯・スタッフ別。空き枠=取り返せない売上 |
| ＋追加#3 | リピート率/継続率 | 30/60/90日再来店率・新規/既存比率 |
| ＋追加#4 | 顧客LTV/VIPランク | 上位2割が売上8割。関係性トリガー(誕生日/記念日/久々来店) |
| ＋追加#5 | 指名率 | スタッフ価値の先行指標。スタッフ別売上と必ずセット |

**毎日見る常設は5画面に集約**(残り2機能は内包/委譲)。優先順に実装する。

# 画面① 経営TOP(最優先・毎日10秒)

```
┌──────────────────────────────┐
│ ◆ 今月の経営(必須4指標)        │
│ 売上    ¥1,842K / 目標 ¥2,500K │
│ 利益    ▲¥38K(暫定)           │
│ 損益分岐 まで ▲¥338K           │
│ 着地予測 売上¥2,210K 利益¥10K  │
├──────────────────────────────┤
│ ◆ KPI(これだけ・異常値で色)    │
│ 本日売上 ¥98K │ 目標進捗 74%   │
│ 次回予約率 71%↗│ DM→予約 23%   │
├──────────────────────────────┤
│ ⚠ 今日の一手(AI・一行指示)     │
│ 🔴 田中様 周期2.3倍・離脱圏     │
│    → 外舘に対応を指示 [指示]    │
│ 🟡 佐藤様 4条件達成・今日が最適 │
│    → サブスク提案を担当へ共有   │
│ [離脱予兆センターへ(7名)]       │
└──────────────────────────────┘
```

| 項目 | 計算・データソース |
|---|---|
| **今月売上/利益/損益分岐/着地予測**(必須) | dashboard_daily.month_sales / month_profit_est(売上×(1−変動費率)−固定費) / breakeven_amount / month_forecast。固定費未設定→「設定待ち」 |
| 本日売上/目標進捗 | 当日visits軽量COUNT / month_sales÷target |
| 次回予約率(追加#1) | 当月 next_booking_made率(前月比矢印) |
| DM→予約転換率(LINE唯一の昇格指標) | scenario_outcomes.booking_within_14d ÷ sent。LINE API紐付け前は暫定値+「概算」表示 |
| 今日の一手(AI Warning) | **決定論ルール・LLM不使用**。離脱圏/サブスク最適日/指名偏り等を**一行の行動指示**で。各行は[担当に指示/共有]の導線まで(**LINE送信などの現場操作はスタッフアプリの責務**でここには置かない)。気づき・グラフは置かない |

設計判断: KPIは4枠固定(増やさない)。経営者は数字オタクでなく異常値だけ拾えればよい、を画面契約にする。

# 画面② 離脱予兆センター(最重要・「分析」でなく「予兆」)

失客レポート(過去)ではなく、**まだ間に合うタイミングで飛ぶアラート**。

```
優先度順(churn_score×LTV):
┌ 山田様 B型 危険度0.82 🔴       │
│ 周期2.3倍経過・前回HC見送り     │
│ 推定損失 年¥336K(客単価×頻度)  │
│ → 外舘さんから S-L-23(気遣い)  │
│   [担当スタッフに対応を指示]    │
│   最終来店 58日前              │
└──────────────────────────────┘
```

| 表示 | ロジック |
|---|---|
| 予兆リスト | dashboard_daily.churn_danger_ids → 優先度=churn_score×0.7+LTV正規化×0.3 |
| 推定損失額 | 客単価×想定年間来店回数(失客の金額インパクトを可視化=高単価サロンの出血) |
| 推奨アクション/担当/LINE | churn_reason別決定論マップ+affinity最高者+ScenarioSelector上位1件(churn販売停止済) |
| 早期警報 | cycle_over_1_5発生推移(離脱の最上流) |

# 画面④ スタッフ分析(管理者専用・売上単体禁止)

3名カード(五十音順・**順位/合計/平均比較なし**)。レビュー指摘どおり売上は必ず指名率・リピート率と同居。

```
┌ 亀山さん ─────────────────────┐
│ 売上 ¥486K │ 指名率 58% │ リピート 64%│
│ 強み: D型成約・理論提案         │
│ 伸びしろ: 実行率の週ムラ        │
│ パターン活用率 87%             │
└──────────────────────────────┘
```

| 表示 | 注記 |
|---|---|
| 売上×指名率×リピート率(3点セット) | **売上単体表示を型で禁止**(押し売り評価の回避)。指名率=追加#5 |
| 強み/伸びしろ | 店舗平均±10pt判定(「苦手」表記禁止) |
| 連携 | スタッフ本人画面(Staff KPI v2.0)とは別経路。本画面はManager専用(staff=403) |

# 画面⑤ 稼働率(席・予約枠・追加#2)

```
曜日×時間帯ヒートマップ + スタッフ別稼働
火 14-16時 □□□□ 20%(空き枠=取り返せない売上)
→ この枠にDM自動化のターゲティング/シフト調整
```

| 表示 | データソース |
|---|---|
| 枠稼働率 | bookings ÷ capacity(business_settings.seat_capacity・曜日時間帯別) |
| スタッフ別稼働 | staff_id別の予約時間占有率 |
| 連携 | 低稼働枠→DM自動化のターゲティング示唆/シフト最適化 |

# 内包・委譲(常設タブにしないもの)

| 機能 | 行き先 |
|---|---|
| 売上予測(必要来店数・必要客単価) | 経営TOPの着地予測+[詳しく]展開(独立画面にしない) |
| 成功パターン分析(昇降格承認) | revisions承認は経営TOPのWarning経由+Audit Reportへ。常設タブから外す |
| 各種レポート | Audit Report v1.0(ボタン・ON/OFF・7日キャッシュ) |
| 店舗比較/多店舗 | 2店舗目確定月まで非表示(コードは存在・UIのみ封印) |

# DB差分(Master Schema v2.0 = W19)

```
dashboard_daily 追加列:
  rebooking_rate / dm_to_booking_rate / repeat_30 / repeat_60 / repeat_90 /
  new_ratio / nomination_rate(store) / month_profit_est
  vip_customer_ids JSONB / relation_triggers JSONB(birthday/anniversary/dormant)
  occupancy JSONB(曜日×時間帯×staff)
business_settings 追加列: seat_capacity JSONB(曜日時間帯別の席数) / variable_cost_rate
brain_staff: スタッフ別 nomination_rate/repeat_rate は集計でdashboard_cacheへ(列持たず)
```

集計は全て nightly-dashboard(DashboardAggregator拡張)で生成→画面は読むだけ(1画面1リクエスト)。


# 実装順(残す機能を優先・Claude Code)

```
MD-1 経営TOP(必須4指標+KPI4+AI Warning一行指示)         ← 最優先・単独で価値が立つ
MD-2 離脱予兆センター(予兆アラート+アクション+推定損失)
MD-3 顧客資産(VIP/関係性トリガー/リピート率)
MD-4 スタッフ分析(3点セット・売上単体禁止のAPI契約)
MD-5 稼働率(seat_capacity設定UI込み)
前提: W19マイグレーション+DashboardAggregator拡張(nightly先行)
受入: 各画面1リクエスト/AI Warningが一行行動指示(気づき列挙の不在テスト)/
      staff-analyticsに売上単体・順位フィールド不在/KPIが4枠固定/固定費未設定でprofit=null
```
