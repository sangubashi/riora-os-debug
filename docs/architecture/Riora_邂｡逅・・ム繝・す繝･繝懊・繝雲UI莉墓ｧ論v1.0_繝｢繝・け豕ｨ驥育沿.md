# Riora 管理者ダッシュボード UI仕様 v1.0(モック注釈版)

**株式会社martylabo / Salon Riora**
作成日: 2026-06-12
位置づけ: アップロードされたUIモックを土台に、**ビジュアルは採用・情報設計はManagement Dashboard v2.0に準拠**させた実装用UI仕様。Claude Codeはこの注釈どおりにモックを修正して実装する。

## 0. 結論(モックからの変更サマリー)

| 領域 | モック現状 | v1.0仕様(変更) |
|---|---|---|
| ビジュアル(配色/余白/タイポ/カード/キャラ) | 良好 | **そのまま採用** |
| サイドバー | 10項目(死蔵タブ多数) | **6項目に削減** |
| KPI帯 | 本日売上/次回予約率/店販率/失客注意/新規 | **必須4指標(売上・利益・損益分岐・着地)を主役化**+KPI4枠 |
| スタッフ別 | 売上/次回予約率/店販率 | 店販率→**リピート率**に差替(押し売り評価回避) |
| LINE/年齢層別 | 独立カード | LINEは「DM→予約転換率」1指標化・年齢層別は撤去 |

## 1. サイドバー(10→6項目)

```
保持(=Management Dashboard v2.0の5画面+設定):
 ● ダッシュボード      (経営TOP=画面①)
 ● 失客リスク管理      (離脱予兆センター=画面②)
 ● 顧客管理           (顧客資産 LTV/VIP=画面③)
 ● スタッフ分析        (画面④・管理者専用)
 ● 店販分析 → 「稼働率」へ改称  (席・予約枠稼働=画面⑤)
 ● 設定

撤去(死蔵タブ・v2.0の判定どおり):
 ✕ 売上・KPI    → ダッシュボードに内包(独立不要)
 ✕ LINEマーケティング → 削除。「DM→予約転換率」1指標を①KPIへ
 ✕ 予約・カレンダー  → 削除。SalonBoardが予約の唯一の正
 ✕ レポート       → Audit Report(ボタン起動・別画面)へ委譲
```

サイドバー下部の「管理者アカウント(くま)」ブロックはそのまま保持。

## 2. ヘッダー

- 「こんばんは、管理者さん 🌙」+日付 = **そのまま採用**(時間帯挨拶のトーンが良い)
- 右上「AIアシスタント(くま)」: 管理画面では**承認待ち等の通知ハブ**としてのみ使う(称賛・指導の機能は持たせない)。月次で「承認待ち◯件」を出す導線をここに(前回合意: 新規画面を作らず通知ハブ経由で突きつける)

## 3. KPI帯(最重要の修正)

モックの5カードを**2層**にする。上段=経営の必須4指標(主役)、下段=日次KPI4枠。

```
┌── 上段: 今月の経営(必須・大きく) ──────────────┐
│ 売上 ¥1,842K/目標¥2,500K │ 利益 ▲¥38K(暫定)    │
│ 損益分岐 まで ▲¥338K      │ 着地予測 ¥2,210K     │
└──────────────────────────────────────────────┘
┌── 下段: 今日のKPI(異常値で色・4枠固定) ─────────┐
│ 本日売上¥182K(+12%)│次回予約率78%↗│店販率32%│DM→予約23%│
└──────────────────────────────────────────────┘
```

- **追加必須**: 利益・損益分岐・着地予測(モックに不在だった)。固定費未設定の月は利益=「設定待ち」表示
- 「失客注意数3名」カードは**KPIから外し、失客リスク管理セクションのヘッダーバッジ**へ移動(数字単独より、その場で対処できる場所に置く)
- 「新規来店数」は下段KPIの予備枠 or 顧客管理へ。4枠は増やさない(KPIを増やすと見ない画面になる原則)
- DM→予約転換率: LINE API紐付け前は「概算」ラベル

## 4. 売上推移チャート

- 実績(実線)vs目標(破線)+「5/23時点 ¥2,120,000」注釈 = **そのまま採用**(優秀)
- 下部の達成率バー74%も保持。**損益分岐ラインを水平線で1本追加**(目標到達前に黒字転換点が見える)

## 5. スタッフ別パフォーマンス

```
モック: Suzuki 売上¥680K / 次回予約率82% / 店販率34%
仕様 : Suzuki 売上¥680K / 次回予約率82% / リピート率__%
```

- **店販率 → リピート率に差替**(レビュー#4b: 売上単体・店販偏重は押し売り評価を生む)。指名率も追加可
- **管理者専用**(staffロールは403)。スタッフ本人画面はStaff KPI v2.0が担当(別経路)
- カード内に順位・合計・店舗平均比較は出さない(ランキング禁止・v1.1継承)。3名は五十音/固定順

## 6. 失客リスク TOP5(離脱予兆センター入口)

モックの形式(顧客・最終来店・理由タグ・スコア)= **方向は正しい・採用**。追加:
- 各行に**経営判断の導線のみ**: [担当スタッフに対応を指示](=スタッフアプリの該当顧客にフラグを立てる/担当へ通知)。**LINEの送る/送らない・文面確認はここでは行わない**(現場オペレーションはスタッフアプリの責務)。経営側は「誰に手を打つべきか」を見て指示するところまで
- スコアの隣に**推定損失額**(年¥◯◯K)を小さく(高単価サロンの出血を可視化)
- VIPタグ(佐藤様)は維持。churn>0.7は販売系LINEを出さない(エンジン側で担保済)

## 7. 下段3カードの入れ替え

| モック | 仕様 |
|---|---|
| LINE配信パフォーマンス(配信数/開封率/返店率/来店化率) | **撤去**。価値ある「来店化率(=DM→予約)」は③KPIへ昇格済。虚栄指標(配信数/開封率)は載せない |
| 年齢層別 売上構成比 | **撤去**(動けない指標)。空いた枠に**席稼働率ヒートマップ**(曜日×時間帯)を配置=画面⑤の縮小版 |
| 今月のAIインサイト | **保持・ただし一行の行動指示に限定**。「◯◯ニーズ上昇」(気づき)でなく「今日◯◯様に△△を提案」(指示)。[もっと詳しく分析する]=Audit Report起動(コスト確認モーダル経由) |

## 8. くまアイコン(モックのくま)

- 管理者ダッシュボードに**称賛・指導の機能は置かない**(称賛/指導という役割分担の概念は廃止。ツンくま姉さん等のキャラクター運用も廃止)
- ヘッダー右「AIアシスタント」のくまは**通知ハブ兼ブランドの装飾**としてのみ存在。役割は承認待ち件数の月次突きつけ・各種通知の表示に限定し、人格的な声かけはしない
- くまの大きいイラストは装飾として2箇所までに抑える(画面が賑やかになりすぎない範囲)

## 9. 実装メモ(Claude Code)

```
- 技術: モックのコード例(Next.js + Tailwind + Recharts)を踏襲
- データ源: 全カードは GET /api/dashboard/* (dashboard_daily/cache の読取・1画面1リクエスト)
- 配色トークン: モックのピンクベージュをそのまま変数化(#F8F1F3系)。新色を足さない
- 受入: サイドバー6項目/KPI上段4必須+下段4枠固定/スタッフにリピート率(店販率でない)/
        LINE配信・年齢層別カードの不在/AIインサイトが一行指示(気づき列挙の不在テスト)
- UI凍結との関係: 本修正は管理者画面(admin枠)であり、現場アプリのUIロック対象外
```
