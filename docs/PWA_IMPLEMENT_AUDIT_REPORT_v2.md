## 1. スプラッシュ画面の正確な実装状況

### 1-1. 実際に動いているスプラッシュ(ルート`/`)

```
app/page.tsx (ルート "/" )
  → <SplashScreen />（src/components/SplashScreen/index.tsx）を表示
  → 背景画像 /splash/splash-bg.png + アニメーションするローディングドット3つ
  → 裏でSupabaseセッションを確認(Safari ITP対策で3秒タイムアウト付き)
  → 最低0.8秒はスプラッシュを表示し続けてから
    - セッションあり → /phase1 へ router.replace
    - セッションなし → /login へ router.replace
```

これは**実装済みで、正しく動作している**。ユーザーが説明した「起動後に
アドレスとパスワード入力画面に遷移する」という体験は、まさにこのコンポーネントが
担っている。前版はこれを見落としていた。

### 1-2. もう1つの`/splash`ルート(未使用・別物)

```
app/splash/page.tsx → <SplashIntroScreen />（src/components/splash/SplashIntroScreen.tsx）
  → 画像 /riora-os/wallpaper.png を全画面表示するだけ
  → 遷移ロジック無し(押しても進まない・自動遷移も無い)
  → grep確認: このURLへnavigateする箇所はコード上どこにも無い
    (router.push/replace・<Link>いずれも0件。app/ClientShell.tsxの
     PUBLIC_PATHS配列に名前があるのみ＝アクセスされた場合に認証リダイレクトの
     対象外にする、という安全策としてのみ機能)
```

**つまり`/splash`ルートは実質デッドコード**。前版で見つけていた
`SplashScreen`との名前の類似（`SplashScreen` vs `SplashIntroScreen`）が
混同の一因。実際にアプリが使っているのはルート`/`直下の`SplashScreen`の方で、
`/splash`という名前のルートではない。

### 1-3. 【重要・新規発見】PWAとして起動した場合はこのスプラッシュを一切通らない

```
public/manifest.json の start_url: "/phase1"
```

PWAとしてホーム画面から起動すると、ブラウザは**manifestのstart_urlへ直接遷移する**。
つまりルート`/`（スプラッシュを表示するpage.tsx）を経由せず、`/phase1`へ
いきなり到達する。

その結果、ホーム画面アイコンからの起動時に実際に表示されるのは：

```
1. まず app/ClientShell.tsx の認証確認中ローディング画面
   （背景 #FFF8F7、"読み込み中…" というテキストのみ・画像なし）
2. 認証確認完了後、未ログインなら /login へリダイレクト
```

**つまり、ブラウザでURL(`/`)から開いたときは「画像付きのきれいなスプラッシュ→
ログイン」という体験になるが、ホーム画面から起動したPWAとしてのアクセスでは
その画像スプラッシュを一切経由せず、無地の"読み込み中…"テキストが一瞬
表示されるだけになっている。** これがPWA化の観点で最も実害の大きい差分であり、
前版で「スプラッシュが無い」と誤認した実質的な原因でもある(コード自体は
存在するが、PWA起動経路には配線されていなかった)。

## 2. PWA化の観点での改善点

| # | 改善点 | 内容 | 優先度 |
|---|---|---|---|
| 1 | **start_urlをスプラッシュ経由にする** | `manifest.json`の`start_url`を`/phase1`→`/`に変更する。これだけでPWA起動時も既存のスプラッシュ画像+セッション確認+自動遷移ロジックがそのまま使われるようになる。**新規実装ではなく設定変更1行で解決できる** | 高 |
| 2 | **iOS向けネイティブ起動画像(apple-touch-startup-image)は別問題として残る** | 上記1を直しても、`SplashScreen`はReactコンポーネント＝JSが読み込まれて実行されて初めて表示される。iOSのPWA(ホーム画面追加アプリ)は、JSが読み込まれるまでの間は白い画面が一瞬映る仕様があり、これを避けるための`apple-touch-startup-image`(OSレベルの起動画像、JS実行前に表示される)は現状も未実装。1を直した上でのゆるやかな改善項目として扱うのが妥当 | 中 |
| 3 | **`/splash`ルート(SplashIntroScreen)の扱いを決める** | 現状デッドコード。今回は「削除するかどうかの判断」は範囲外(実装禁止フェーズのため)だが、次フェーズで「削除」「オンボーディング用に転用」のどちらにするか意思決定が必要 | 低 |
| 4 | **スプラッシュの最短表示時間(0.8秒)とService WorkerのCache First戦略の整合** | `SplashScreen`表示中に`/splash/splash-bg.png`が未キャッシュだと初回表示が遅れる可能性。現状の`sw.js`はアプリシェルとして`/phase1`等はキャッシュ対象にしているが、`/splash/splash-bg.png`はキャッシュリストに入っていない(要追加検討) | 低〜中 |
| 5 | **オフライン時の遷移**(前版から継続) | スプラッシュのセッション確認がオフラインで失敗した場合、3秒タイムアウトで`/login`に落ちる仕様。オフライン時にいきなりログイン画面を出すより、正本§3の「オフラインです」という優しい案内を先に出す方がPWAとしては自然 | 中 |
