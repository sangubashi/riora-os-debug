# 既存重複顧客 統合方針調査

**ステータス: 調査のみ・コード変更禁止・migration禁止・commit禁止・push禁止・deploy禁止**
**作成日: 2026-07-20**
**前提**: Pass A+C本番反映(`714db3b`)後、新規重複の発生は大幅に抑止された。本書は**既存の重複顧客の整理方針**を検討する。
**関連する既存設計**: `docs/CUSTOMER_DUPLICATE_MANAGEMENT_V1.md`(検出・統合UIの先行設計、本調査はこれを実データで検証・補強する位置づけ)

## 0. 結論の先出し

```
重複候補は12グループ・54件 → 14グループ・58件に増加していた(実データ再集計)。
増加分の2グループ(吉田雅・熊谷まりあ)は、本日Pass A+C本番投入後の初回Import
(05:40実行)で新規作成されたばかりの重複である。Pass A+Cは「同一人物が複数回来店
するたびに新規顧客が量産される」パターン(パターンB)は防止できたが、「visit実績が
無い既存スタブ顧客と同名の新規来店」パターン(パターンA)は防止できていない
(設計時から把握していた既知の限界が実際に再現した)。

58件中、安全に自動統合できると判定できるのは30件(パターンB・6グループ)。
残り28件(パターンA・8グループ)は視認できる裏付けが薄く、管理者確認が必要。
明確に「統合禁止」と判定すべきケースは実データ上は見つからなかった。

推奨実装方式は「統合候補キュー」(案2)。理由は既存設計書(CUSTOMER_DUPLICATE_
MANAGEMENT_V1.md)がUI/データ移行ルールを既に定義済みであること、およびパターンAの
重複が今後も発生し続ける以上、一回限りのSQLではなく継続運用できる仕組みが必要なため。

なお、既存設計書には無かった技術的リスクを1点新たに発見した: brain_visits.
visit_count_atは顧客単位の原子的連番(RPC採番)のため、統合時にvisit_count_atの
再採番を行わないと、統合後の顧客に「1件目」が複数件重複するデータ不整合が発生する。
```

### 1.3 【重要な新規発見】Pass A+Cは全パターンの重複を防げていない

「吉田雅」「熊谷まりあ」の2グループは、**本日のPass A+C本番投入後、最初のImport実行(05:40)で新規に作られた重複**である。原因を特定した:

```
1. 対象氏名は既にvisit=0のスタブ顧客として登録済み(reservationImportPipeline経由と推定)
2. 新しい来店行が到着 → findNameCandidates()は氏名一致でこのスタブを候補として検出
3. resolveByVisitProximity()はvisit実績が無い候補を除外する(パターンAを意図的に
   除外する設計。docs/CSV_IMPORT_DUPLICATE_ROOT_CAUSE_AUDIT_V1.md §2.2参照)
   → withVisits.length===0 → 'no_visit_history'を返す
4. 旧Pass N(氏名+初回来店日)フォールバックへ: スタブのfirstVisitDateがNULLの場合、
   `sole?.firstVisitDate != null`の条件を満たさずマッチ失敗
5. 候補が存在するため needs_review へ。reviewDecisions未指定のため既定'new'
   → 新規重複顧客が作成される
```

これは`docs/DUPLICATE_PREVENTION_IMPLEMENTATION_PLAN.md`の設計時点で「パターンA
(視認できるvisit証拠が薄いグループ)は自動統合の対象外」と明記されていた既知の限界が、
実際に本番で再現したものであり、Pass A+C自体の不具合ではない。ただし**放置すると
毎週のCSV取込のたびにパターンAの重複が増え続ける**ため、本調査の統合方針とは別に、
「スタブ顧客(reservationImportPipeline由来のvisit=0レコード)への対処」を今後の
別課題として明記しておく(本書のスコープ外・コード変更は行っていない)。

| **A: 安全に自動統合可能** | **24件**(30件中、主レコード6件を除く) | 6(パターンB全て) | 完全氏名一致・来店日非衝突・visit実績あり。全レコードが1visitずつ、来店日が重複しない。担当スタッフのばらつきは「指名なし来店」の自然な証跡であり誤統合の兆候ではない(実データで再確認済み) |
| **B: 管理者確認後に統合** | **28件** | 8(パターンA全て) | 1件のみ実来店の裏付けがあり、残りはvisit=0のスタブ。表記ゆれ(スペース有無)も混在。スタブが本当に同一人物の重複か、単なる未使用の空レコードかを機械的に区別できない |

**A区分(24件)の主レコード選定案**: 各グループ内で「visit件数が最も多い」または同数の場合「created_atが最も早い」レコードを主(生き残る)候補とする。実データでは全レコードが1visitずつのため後者の基準が実質的な決定要因になる。

## 3. 統合した場合の影響分析

### brain_customers

- A区分24件が`deleted_at`設定(論理削除)により非表示化される想定(既存の`deleted_at`列を利用。物理削除は不可逆なため非推奨)
- 主レコード側の`first_visit_date`は「グループ内で最も古い日付」に更新するのが正しい(現状は各レコードが自分の1回の来店日をそのまま`first_visit_date`として持っており、統合後は最古の日付が本来の初回来店日になる)

### brain_visits

- 統合元(消える側)の`customer_id`を統合先(生き残る側)へ`UPDATE`する(`CUSTOMER_DUPLICATE_MANAGEMENT_V1.md` §9.1で既定義)
- **【本調査で新規発見した技術的リスク】`visit_count_at`の再採番が必須**。`brain_visits.visit_count_at`は`supabase/migrations/20260709_insert_visit_with_sequence_rpc.sql`により**顧客単位でMAX+1により原子的に採番**される設計であり、各分裂レコードは独立して「自分にとっての1件目」として`visit_count_at=1`を持っている。単純に`customer_id`だけを付け替えると、統合後の顧客が`visit_count_at=1`のvisitを複数件持つ状態になり、来店回数バッジ・「○回目のご来店」表示等に不整合が生じる。統合処理では**来店日の昇順に`visit_count_at`を1から振り直す**追加ステップが必須(既存の`CUSTOMER_DUPLICATE_MANAGEMENT_V1.md`にはこの言及が無く、本調査で補強が必要と判明した点)

### KPI

- `DashboardAggregator.ts`は`brain_visits.customerId`から動的にLTV・来店回数を集計する設計のため、`customer_id`付け替えが完了すれば**追加の集計テーブル更新は不要**(`CUSTOMER_DUPLICATE_MANAGEMENT_V1.md` §9.1で既に確認済みの設計特性)
- 経営TOPの「総顧客数」は統合によりA区分だけで24件減少する。これは**データ品質改善の結果であり不具合ではない**が、運用者への事前説明が無いと「顧客が減った」と誤解されるリスクがある

### 離脱予兆

- `ChurnRiskEngine`は「visit≥2件」の顧客のみを対象とする設計。パターンB(6グループ)は現状**各レコードが1visitずつのため、そもそも離脱予兆判定の対象になっていない**(構造的に対象外)。統合後は7visits等を持つ通常の顧客として初めて離脱予兆判定の対象に入る。**統合は離脱予兆の検出精度を新たに向上させる効果がある**(退行ではなく機能追加に近い)

### ホームケア通知

- `retailProductCounts`(商品購入回数の集計)も`customer_id`単位のため、統合により購入頻度が正しく積算される。現状は分裂により「初回購入」に見えている行が、統合後は「N回目の購入」として正しく`homecare_checkin`/`homecare_replenish`の判定対象になる可能性がある

### スタッフ分析

- `staff-analytics`は`reservations`/`brain_visits`の`staff_id`起点で集計するため、直接的な計算式の変更はない。ただし「担当customer数」等の指標がある場合、統合によりユニーク顧客数が減るため数値が変動する(パターンBは担当が来店ごとに違うため、統合により「複数スタッフで共有する1顧客」として現れるようになる → 特定スタッフの成績を見かけ上変動させる可能性がある点は関係者へ事前共有が必要)

## 4. 統合実行の実装方式比較

| 観点 | 案1: 管理画面から手動統合 | 案2: 統合候補キュー | 案3: SQL一括統合 |
|---|---|---|---|
| **工数** | 中〜高。`CUSTOMER_DUPLICATE_MANAGEMENT_V1.md`の比較画面・統合フローUI一式を新規実装 | 中。案1とほぼ同じ実行ロジックを再利用しつつ、`brain_customers.merge_status`(confirmed/candidate/unmatched)等の永続的な状態管理を追加 | 低。SQLスクリプト1本で完結 |
| **リスク** | 中。1件ずつ人手確認するため誤統合リスクは低いが、実装量が多く不具合混入の余地もそれなりにある | 中。案1と同等の確認プロセスを維持しつつ、キューの陳腐化・二重処理などの状態管理リスクが追加される | **高**。人手確認なしの一括処理のため、想定外のケースが1件でも混じっていた場合に大量データへ同時に影響する。§3で発見した`visit_count_at`再採番等の複雑な処理をSQLで正確に書く難易度も高い |
| **rollback性** | 高(`deleted_at`論理削除+統合ログを残せば個別に復元可能。ただし復元手順自体は実装が必要) | 高(案1と同様。加えてキュー自体が「何を統合待ちにしたか」の記録になる) | 中〜低(1トランザクションで全件を囲めば全体ロールバックは可能。ただし個別の1件だけを後から取り消すのは、事前にどの操作がどのレコードに影響したかを別途記録していない限り困難) |
| **既存資産の活用** | `CUSTOMER_DUPLICATE_MANAGEMENT_V1.md`のUI設計をそのまま実装対象にできる | 同左+§5(本調査)のA/B/C分類をそのままキューの初期状態として投入できる | 既存設計のUI部分は使わない(比較画面での目視確認プロセスを省略するため) |
| **今後の運用適合性** | 低〜中。都度手動で「重複を探しに行く」運用になり、パターンAの新規発生(§1.3)に対して受動的 | **高**。CSV取込のたびに新規重複が検出された場合、自動的にキューへ積まれる設計にできれば、§1.3のような新規発生にも継続的に対応できる | 低。一度きりの解消にしかならず、パターンAが今後も発生し続ける前提と噛み合わない |

## 5. 推奨案

**案2(統合候補キュー)を推奨する。**

理由:

1. **§1.3で判明した通り、パターンAの重複は現在も発生し続けている。** 一回限りのSQL一括処理(案3)では今回の58件を解消しても、来週には再び新しい重複が積み上がる。継続的に候補を受け付けられる仕組みが必要
2. `CUSTOMER_DUPLICATE_MANAGEMENT_V1.md`が既にUI設計・データ移行ルールの大枠を定義済みであり、ゼロから設計する必要がない(工数面で案1と大差ないが、キューという状態管理を足すだけで運用性が大きく上がる)
3. §3で発見した`visit_count_at`再採番のような複雑な処理は、人手確認を伴う実行フロー(案1・案2)の方が、SQL一括処理(案3)よりも安全に実装・検証しやすい
4. A区分(24件)についても、影響範囲(§3)を鑑みると「機械的に正しそうに見えても、要確認1クッションを挟む」運用が無難であり、案2のキューに「高確度(A区分)/要確認(B区分)」のラベルを付けて表示する形が、今回の調査結果(§2の分類)をそのまま活かせる

**次のアクション(未実施・本調査のスコープ外)**: 案2の実装に進む場合、`CUSTOMER_DUPLICATE_MANAGEMENT_V1.md`の統合フロー設計(§4〜§9)に、本調査で発見した`visit_count_at`再採番ステップを追加した上で、別途実装承認を得ること。またパターンAの新規発生を止めるための追加対応(スタブ顧客の扱い見直し等)も、統合機能とは別に検討が必要。

## まとめ

1. 重複候補は58件(14グループ)に増加。うち2グループはPass A+C本番投入後に新規発生したもので、パターンA(視認できるvisit証拠が薄い重複)がPass A+Cの防御範囲外であることが実データで確認された
2. A区分(安全に自動統合可能)24件・B区分(管理者確認後)28件・C区分(統合禁止)0件
3. 統合の影響は`brain_customers`/`brain_visits`が中心。KPI・離脱予兆・ホームケア通知は間接的にデータ精度が向上する方向。ただし`visit_count_at`の再採番という、既存設計書に無かった技術的対応が必須と判明した
4. 実装方式は案2(統合候補キュー)を推奨。パターンAの重複が今も発生し続けている以上、継続運用できる仕組みが必要なため

本調査ではコード変更・migration作成・commit・push・deployのいずれも行っていません。
